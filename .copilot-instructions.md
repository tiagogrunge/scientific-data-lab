# Scientific Data Lab - Development Guidelines

## Commit Message Format

Generate commit messages following the Conventional Commits specification:

```
type: brief description (50 characters max)

- Detailed explanation of changes
- Use bullet points for multiple related changes
- Keep lines to 72 characters or fewer
- Be specific about what was changed and why
```

### Commit Types

- **feat**: New feature or functionality
- **fix**: Bug fix or issue resolution
- **refactor**: Code reorganization without feature changes
- **test**: Adding or updating tests
- **docs**: Documentation updates
- **style**: Code formatting, linting (no functional changes)
- **perf**: Performance improvements
- **chore**: Dependency updates, build changes

### Examples

```
feat: add CSV validation for malformed data

- Implement column count validation in load_csv()
- Raise ParserError for mismatched column counts
- Add test case for invalid CSV format
```

```
chore: update dependencies for Python 3.14 support

- Update pandas to >=2.1.0 for Python 3.14 compatibility
- Update numpy to >=1.26.0 compatible version
- Add pandas-stubs for type checking support
```

## Code Quality Standards

- Python version: >=3.10 (enforced by mypy)
- Type hints: Required for all function definitions
- Line length: 100 characters (enforced by ruff)
- Imports: Alphabetically sorted, grouped logically
- Tests: Required for all new features

## Version Verification Before Coding

**Before suggesting any code, dependencies, or version updates:**

1. **Check Current Recommended Versions**
   - Verify the current supported versions of Python and major dependencies
   - Reference official sources: Python.org versions page, package PyPI pages, changelog links
   - Avoid suggesting outdated versions even if they appear in training data or boilerplate examples

2. **For Python Runtime**
   - Current target: Python 3.14+
   - Always validate against PEP release schedules and support timelines
   - Check if suggested version has active security/bugfix support

3. **For Dependencies** (pandas, numpy, matplotlib, pytest, ruff, mypy)
   - Verify latest stable release compatible with Python 3.14
   - Check release notes for breaking changes or deprecations
   - Ensure type stubs and annotations are available for the suggested version
   - Reference: package PyPI pages, GitHub release notes, official documentation

4. **For Code Examples**
   - Ensure code patterns reflect current best practices for the version era
   - Avoid deprecated APIs or syntax even if commonly found in older examples
   - Test suggestions against current type checkers (mypy) and linters (ruff)

5. **For CI/CD and GitHub Actions**
   - Verify GitHub Actions runner versions (ubuntu-latest, macos-latest, windows-latest)
   - Check action versions (actions/setup-python@v6, actions/checkout@v6, etc.) for current compatibility
   - Ensure workflow configuration matches current best practices and deprecation notices
   - Reference: GitHub Actions marketplace, action release notes, runner version pages
   - Common items: Python setup action, linting/testing actions, deployment actions

**Examples**: 
- Before suggesting an action, check its marketplace page for the latest stable version and Python compatibility
- Verify that workflow runners support Python 3.14
- Ensure GitHub Actions syntax hasn't deprecated features (e.g., `set-output` was deprecated in favor of environment files)

## Testing

Run before committing:
```bash
pytest tests/ -v
mypy src/ tests/
ruff check src/ tests/
```
